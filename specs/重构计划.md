## 一、计划摘要

本计划描述了 RClick 架构重构的详细实施步骤，目标是实现**瘦 Extension 架构**，将 FinderSync 扩展简化为纯渲染层。

### 实施顺序（v4 - 加入 Phase 0）

| 顺序 | Phase | 优先级 | 风险 | 说明 |
|------|-------|--------|------|------|
| 0 | **Phase 0: 设计差异修复** | **P0** | **低** | **修复设计与实现差异** |
| 1 | Phase 1: 统一 Icon 缓存 | P1 | 低 | 已存在，让 Extension 使用 |
| 2 | Phase 2: IPC 消息安全加固 | P0 | 低 | 新增功能，不影响现有逻辑 |
| 3 | Phase 3: 全盘访问权限 | P1 | 中 | 只做添加，暂不删除旧代码 |
| 4 | **Phase 4: 瘦 Extension 架构** | **P0** | **高** | **核心重构** |
| 5 | Phase 5: Tag 管理优化 | P1 | 低 | Phase 4 的自然延伸 |
| 6 | Phase 6: Extension 重连机制 | P1 | 中 | 依赖 Phase 4 的菜单缓存 |
| 7 | Phase 7: 脚本执行功能 | P1 | 中 | 依赖 Phase 2 的安全消息 |
| 8 | Phase 8: 清理和验证 | P2 | 低 | 统一清理 |

| **Phase 6 风险降为中** | 改用 SwiftData 实时构建，不再需要持久化同步 |

#### 步骤 0.1: `running` 消息携带菜单配置（P0）

**文件**：`RClick/RClickApp.swift`

**修改内容**：
- 第一次 `running` 发送完整 `MenuConfigPayload`
- 后续重试只携带时间戳（`RunningPayload(timestamp:)`），Extension 发现缓存过期再主动 `requestConfig`
- 减少不必要的完整配置构建和 JSON 序列化

**设计理由**：主 App 启动后指数退避重试 3–4 次，如果每次都要构建完整 `MenuConfigPayload`（几十个 action/app），JSON 序列化成本可能导致性能抖动。首次全量 + 后续仅时间戳的方案既保证 Extension 能快速获得配置，又避免重复传输开销。

**当前代码**：
```swift
// RClickApp.swift:169
func sendObserveDirMessage() {
    let directories: [String] = []
    messager.sendRunningNotification(directories: directories)
}
```

**修改为**：
```swift
// 第一次发送：携带完整配置
func sendRunningMessage() {
    let config = buildMenuConfig()
    messager.sendToExtension(.running, data: config)
}

// 后续重试：仅携带时间戳
func sendRunningMessageRetry() {
    let timestamp = Date().timeIntervalSince1970
    messager.sendToExtension(.running, data: RunningPayload(timestamp: timestamp))
}

struct RunningPayload: Codable {
    let timestamp: TimeInterval
}
```

**Extension 端处理**：
```swift
private func handleRunning(_ data: Data?) {
    // 尝试解码完整配置
    if let config = decode<MenuConfigPayload>(data) {
        applyConfig(config)
        return
    }
    // 仅时间戳：如果本地缓存为空，主动请求
    if cachedMenuConfig == nil {
        sendMenuRequest()
    }
}
```

### Phase 6: Extension 重连机制 ⭐⭐ P1

**目标**：解决主 App 已打开但 Extension 未加载的问题

**预计时间**：2-3 小时

**风险**：低

#### 设计原则（必须遵守）

| 原则 | 说明 | 违反后果 |
|------|------|----------|
| **Notification ≠ 状态** | 通知只能当"信号"，不能当"真相" | 丢消息=状态错乱 |
| **单一持久化源** | 业务数据只存 SwiftData，快照实时构建 | 双写一致性风险 |
| **Extension 必须支持主动拉取** | 不能只靠推送 | 无法自愈 |
| **所有通信必须幂等** | 重复消息不能出错 | 重复执行风险 |
| **时间戳防乱序** | 用时间戳判断配置新旧，拒绝旧配置 | 乱序覆盖问题 |

#### 步骤 6.1: 实时构建配置机制（P0）⭐ 核心

**文件**：`RClick/RClickApp.swift`

**修改内容**：
- 添加 `buildMenuConfig()` 方法从 SwiftData 实时构建
- 添加内存缓存和 `lastBuildTime` 用于防抖（1秒内不重复构建）
- `MenuConfigPayload` 添加 `timestamp: TimeInterval` 字段
- 注册 `requestConfig` 消息处理器，调用 `buildMenuConfig()` 发送

**核心设计**：

```swift
// 内存缓存（仅用于防抖）
private var cachedMenuConfig: MenuConfigPayload?
private var lastBuildTime: TimeInterval = 0

/// 构建菜单配置（从 SwiftData 实时构建）
func buildMenuConfig() -> MenuConfigPayload {
    let now = Date().timeIntervalSince1970
    
    // 如果 1 秒内已构建过，直接返回缓存（防抖）
    if now - lastBuildTime < 1.0, let cached = cachedMenuConfig {
        return cached
    }
    
    // 从 SwiftData 读取数据
    let actions = appState.actions.map { $0.toActionMenuItem() }
    let apps = appState.apps.map { $0.toAppMenuItem() }
    let newFiles = appState.newFiles.map { NewFileMenuItem(...) }
    let commonDirs = appState.cdirs.map { CommonDirMenuItem(...) }
    
    let config = MenuConfigPayload(
        timestamp: now,
        actions: actions,
        apps: apps,
        newFiles: newFiles,
        commonDirs: commonDirs
    )
    
    cachedMenuConfig = config
    lastBuildTime = now
    
    return config
}

// 响应 Extension 的 config 请求
func handleConfigRequest() {
    let config = buildMenuConfig()
    messager.sendMenuConfig(config)
}

// 发送 running 消息时携带当前配置
func sendRunningMessage() {
    let config = buildMenuConfig()
    messager.sendToExtension(.running, data: config)
}
```

**优势**：
- 单一持久化源（SwiftData），无一致性风险
- 实时构建毫秒级，无额外存储开销
- 时间戳防乱序，无需维护递增版本号

#### 步骤 6.2: 主动请求机制（P0）⭐ 核心

**文件**：`FinderSyncExt/FinderSyncExt.swift`

**修改内容**：
- 添加 `sendMenuRequest()` 方法
- `init()` 中启动时发送请求
- `menu(for:)` 中缓存为空时触发请求
- 接收更新时检查时间戳（防乱序）

```swift
class FinderSyncExt: FIFinderSync {
    private var currentTimestamp: TimeInterval = 0
    private var cachedMenuConfig: MenuConfigPayload?

    override init() {
        super.init()
        setupMessageHandlers()
        // 启动时主动请求配置
        sendMenuRequest()
    }

    private func handleRunning(_ data: Data?) {
        guard let config = decode<MenuConfigPayload>(data) else { return }
        // 时间戳检查：只接受更新的配置
        guard config.timestamp > currentTimestamp else {
            logger.info("Ignoring outdated config")
            return
        }
        currentTimestamp = config.timestamp
        cachedMenuConfig = config
    }

    private func handleMenuConfig(_ data: Data?) {
        guard let config = decode<MenuConfigPayload>(data) else { return }
        // 时间戳检查
        guard config.timestamp > currentTimestamp else { return }
        currentTimestamp = config.timestamp
        cachedMenuConfig = config
    }

    // 菜单构建：只读缓存
    override func menu(for menuKind: FIMenuKind) -> NSMenu {
        guard let config = cachedMenuConfig else {
            // 缓存为空，触发请求
            sendMenuRequest()
            return NSMenu(title: "RClick (loading...)")
        }
        // 使用缓存构建菜单...
    }
}
```

**文件**：`RClick/RClickApp.swift` 和 `FinderSyncExt/FinderSyncExt.swift`

private var isExtensionOnline: Bool = false

func handleHeartbeat() {
    isExtensionOnline = true
    // 重置超时定时器
    heartbeatTimeoutTimer?.invalidate()
    heartbeatTimeoutTimer = Timer.scheduledTimer(withTimeInterval: 15, repeats: false) { [weak self] _ in
        self?.handleHeartbeatTimeout()
    }
}

// Extension 端
Timer.scheduledTimer(withTimeInterval: 10, repeats: true) { _ in

}

**文件**：`RClick/RClickApp.swift`

**修改内容**：

    // 菜单栏显示"扩展未响应"
    // 用户可点击"重新加载扩展"按钮

// 用户点击"重新加载扩展"时
private func refreshExtension() {
    // 发送 running + menuConfig，尝试重新连接
    sendRunningMessageWithRetry()
}
```

**心跳方向说明**：

| 方向 | 消息类型 | 频率 | 目的 |
|------|----------|------|------|
| Main → Ext | `running` | 启动时 + 重试 3-4 次（指数退避：2s→4s→8s） | 告知 Extension 主程序已启动 |
| Ext → Main | `heartbeat` | 每 10 秒 | 告知主程序 Extension 存活 |

**参数调整说明**：
- **重试次数**：从 6 次减少到 3-4 次，减少用户等待时间
- **重试间隔**：采用指数退避（2s, 4s, 8s），总时长从 30 秒降至 14 秒
- **超时阈值**：从 15 秒调整为 25-30 秒（心跳间隔的 2.5-3 倍），避免误判

func detectDangerousCommands(_ scriptContent: String) -> [String] {
    let dangerousPatterns = [
        "rm -rf /", "rm -rf ~", "sudo",
        "dd if=/dev/zero", ":(){ :|:& };:",
        "> /dev/disk", "mkfs", "newfs"
    ]
    return dangerousPatterns.filter { scriptContent.contains($0) }
}
```

**2. 执行前用户确认**：
```swift
func confirmScriptExecution(script: RCScript, files: [String]) -> Bool {
    let alert = NSAlert()
    alert.messageText = "确认执行脚本？"
    alert.informativeText = """
        脚本名称：\(script.name)
        脚本内容预览：\(script.scriptContent.prefix(200))...
        将处理 \(files.count) 个文件
        """
    return alert.runModal() == .alertFirstButtonReturn
}
```

**3. 执行超时控制（后台线程）**：
> ⚠️ **必须在后台线程调用**（`async/await` 或 `DispatchQueue.global`），`process.waitUntilExit()` 在主线程会阻塞界面。
```swift
func executeScriptWithTimeout(_ script: String, files: [String], timeout: TimeInterval = 30) async throws {
    try await withCheckedThrowingContinuation { continuation in
        DispatchQueue.global(qos: .userInitiated).async {
            let process = Process()
            process.executableURL = URL(fileURLWithPath: "/bin/bash")
            process.arguments = ["-c", script, "--"] + files
            
            let pipe = Pipe()
            process.standardOutput = pipe
            process.standardError = pipe
            
            do {
                try process.run()
                
                DispatchQueue.global().asyncAfter(deadline: .now() + timeout) {
                    if process.isRunning { process.terminate() }
                }
                
                process.waitUntilExit()  // 在后台线程阻塞是安全的
                continuation.resume(returning: ())
            } catch {
                continuation.resume(throwing: error)
            }
        }
    }
}
```
