# RClick 架构重构设计

> 最后更新：2026-03-21
> 状态：设计文档
> 架构方向：**瘦 Extension**（Slim Extension）+ **全盘监听** + **按需请求权限**

---

## 一、执行摘要

本重构计划的核心目标是构建**瘦 Extension 架构**，将 RClick 的 FinderSync 扩展简化为纯渲染层，所有业务逻辑和数据管理集中在主程序中。

### 职责划分

| 职责 | Extension | Main App |
|------|-----------|----------|
| 监听范围 | ✅ 全盘（/Users/ + 外接磁盘） | - |
| 数据读取 | ❌ 不读取 | ✅ SwiftData |
| 菜单构建 | ❌ 只渲染 | ✅ 构建配置 |
| 业务逻辑 | ❌ 无 | ✅ 全部 |
| 权限请求 | ❌ 不处理 | ✅ 按需请求 |
| Bookmark 缓存 | ❌ 不处理 | ✅ 缓存授权文件夹 |
| 图标加载 | ✅ 根据路径 | ✅ 预加载缓存 |

**核心原则**：Extension 只做 3 件事——监听文件目录、渲染菜单、发送事件。

### 设计特性

| 特性 | 说明 | 状态 |
|------|------|------|
| 瘦 Extension | Extension 不读取数据，只负责渲染 | ✅ 已设计 |
| 全盘监听 | 默认监听 `/Users/`，外接磁盘自动添加 | ✅ 已设计 |
| 按需请求权限 | 菜单点击后如需权限，由主程序请求 | ✅ 已设计 |
| Bookmark 缓存 | 主程序缓存用户授权的文件夹 bookmark | ✅ 已设计 |
| 类型安全协议 | 使用枚举替代字符串 | ✅ 已设计 |
| Notification + 协议层 | 基于 DistributedNotificationCenter + Codable 协议封装 | ✅ 已设计 |
| 消息 ID | 每个消息带 UUID，支持追踪和去重 | ✅ 已设计 |
| 统一 Icon 缓存 | 共享模块统一管理 | ✅ 已设计 |
| 递增 Tag | 避免随机数冲突 | ✅ 已设计 |

### 预期收益

- Extension 内存使用 < 20MB（无 SwiftData 依赖）
- 菜单渲染时间 < 50ms
- 代码行数减少约 30%
- 维护成本降低约 40%
- 用户体验提升：无需手动配置许可目录

## 二、当前问题总结

### 2.1 架构层面

- **重复实现**: Extension 有 `IconCache`，Main App 有 `IconCacheManager`
- **数据模型冗余**: RCAction/ActionMenuItem、OpenWithApp/AppMenuItem 需要转换
- **通信协议简陋**: `MessagePayload.action` 使用 String 类型，缺乏类型安全
- **转换层复杂**: 需要在 SwiftData 模型和 MenuItem 模型间转换

### 2.2 性能问题

- **频繁 JSON 编解码**: 每次菜单更新都需要通过 UserDefaults 编解码
- **Tag 管理低效**: 使用随机数生成 tag，可能冲突导致性能问题
- **无缓存机制**: 图标每次都重新加载

### 2.3 数据一致性

- **同步延迟**: 配置更新需要等待通知和 UserDefaults 刷新
- **转换错误风险**: 模型转换可能丢失数据或引入 bug

### 2.4 用户体验

- **Main App 依赖**: Extension 在 Main App 关闭时无法工作
- **更新延迟**: 配置更新实时性差（依赖通知 + 数据转换）

## 三、设计目标

### 3.1 核心原则

1. **简洁性**: 减少重复代码，统一 Icon 缓存实现
2. **类型安全**: 使用枚举替代字符串，编译时检查
3. **性能**: 优化 Tag 管理，使用递增 ID；图标预加载缓存
4. **可靠性**: 错误处理完善，日志记录清晰

### 3.2 架构选择：瘦 Extension

Extension 只做 3 件事：

1. **接收消息** - 从主程序接收菜单配置、刷新指令
2. **渲染菜单** - 根据收到的配置渲染菜单（不读任何数据源）
3. **发送事件** - 用户点击后发送操作数据给主程序

**所有业务逻辑在主程序完成。**

### 3.3 核心改进（按优先级）

| 优先级 | 改进项 | 预计时间 | 风险 |
|--------|--------|----------|------|
| P0 | 类型安全通信协议 | 1-2 小时 | 低 |
| P0 | 瘦 Extension 架构 | 2-3 小时 | 中 |
| P1 | 优化 Tag 管理 | 30 分钟 | 极低 |
| P1 | 统一 Icon 缓存 | 1 小时 | 低 |
| P2 | 文档翻译/注释 | 1 小时 | 无 |

### 4.2 新架构职责划分

#### Main App 职责

- **数据管理**: 读取 SwiftData，构建菜单配置
- **业务逻辑**: 处理所有动作执行（复制路径、打开应用等）
- **配置推送**: 数据变更时发送完整菜单配置给 Extension
- **事件处理**: 接收 Extension 的点击事件，执行对应操作
- **权限请求**: 按需请求文件访问权限（NSOpenPanel）
- **Bookmark 缓存**: 缓存用户授权过的文件夹 bookmark 数据

#### Extension 职责

- **文件监听**: 监听全盘文件变化（默认 `/Users/`，自动检测外接磁盘）
- **消息监听**: 监听来自主程序的配置消息
- **菜单渲染**: 根据配置渲染 NSMenuItem
- **事件转发**: 用户点击后发送事件给主程序
- **心跳发送**: 定期发送心跳确认存活

### 4.3 Extension 监听范围

```swift
// FinderSyncExt.swift
override init() {
    super.init()

    // 默认监听 /Users/
    let usersDir = URL(fileURLWithPath: "/Users/")

    // 检测并添加外接磁盘
    let volumeURLs = FileManager.default.mountedVolumeURLs(
        includingResourceValuesForKeys: [.isBrowsableKey],
        options: [.skipHiddenVolumes]
    ) ?? []

    var allURLs = [usersDir]
    for volume in volumeURLs {
        if volume.path != "/" {  // 排除根目录
            allURLs.append(volume)
        }
    }

    FIFinderSyncController.default().directoryURLs = Set(allURLs)

    logger.info("Monitoring directories: \(allURLs.map { $0.path })")

    setupMessageHandlers()
    sendHeartbeat()
}
```

#### 监听策略

- **默认监听**: `/Users/`（用户主目录）
- **外接磁盘**: 自动检测并添加所有可浏览的挂载卷
- **网络卷**: 不主动添加（由用户授权后按需访问）

## 八、附录：Extension 重连方案

### 8.1 问题背景

FinderSync Extension 由 macOS 系统控制加载时机，可能出现**主 App 已打开但 Extension 未加载**的情况，导致右键菜单不显示。

### 8.2 核心设计原则

| 原则 | 说明 | 违反后果 |
|------|------|----------|
| **Notification ≠ 状态** | 通知只能当"信号"，不能当"真相" | 丢消息=状态错乱 |
| **必须存在状态快照** | 否则无法恢复一致性 | 重启后数据丢失 |
| **Extension 必须支持主动拉取** | 不能只靠推送 | 无法自愈 |
| **所有通信必须幂等** | 重复消息不能出错 | 重复执行风险 |

### 8.3 架构：推 + 拉混合模式

```
纯推送（之前）：Main App → 发送消息 → Extension（靠运气接收）
推拉混合（升级）：Main App ←→ 发送/请求 ←→ Extension（可主动拉取）
```

### 8.4 四层防护策略

| 层级 | 机制 | 作用 | 优先级 |
|------|------|------|--------|
| L1 | **状态快照** | 主 App 保存最后菜单状态（真相源） | P0 |
| L2 | **心跳重试** | 主 App 启动后定期发送 running | P1 |
| L3 | **主动请求** | Extension 启动/过期时主动请求 | P0 |
| L4 | **自动重连** | 心跳超时后自动恢复 | P1 |
| L5 | **手动刷新** | 用户菜单触发 | P2 |

### 8.5 能力评估

| 场景 | 之前 | 升级后 |
|------|------|--------|
| Extension 随机重启 | ❌ | ✅ |
| Notification 丢失 | ❌ | ✅ |
| 主 App 先启动 | ⚠️ | ✅ |
| Extension 后启动 | ⚠️ | ✅ |
| Finder 延迟加载 | ❌ | ✅ |
| 状态一致性 | 🟡 弱 | 🟢 接近强一致 |

## 九、设置界面重构设计

### 9.1 设计目标

重新设计设置界面的"通用设置"部分，采用 macOS 系统设置风格（分组卡片式布局），并将"折叠上下文菜单"开关从通用设置移到每个分类里面。

### 9.2 通用设置新布局

通用设置采用三组卡片式布局：

#### 第一组：主要控制

| 项目 | 类型 | 说明 |
|------|------|------|
| 启用「RClick」 | Toggle 开关 | 主开关，控制扩展启用/禁用 |
| 登录时启动 | Toggle 开关 | 系统登录时自动启动 RClick |

#### 第二组：权限

| 项目 | 类型 | 说明 |
|------|------|------|
| Finder 扩展 | 状态显示 | 只读，带绿色勾选图标表示已启用 |
| 完全磁盘访问权限 | 按钮 | 点击跳转到系统设置请求权限 |
| 辅助功能 | 按钮 | 点击跳转到系统设置请求权限 |

#### 第三组：设置管理

| 项目 | 类型 | 说明 |
|------|------|------|
| 备份 | 按钮组 | 导出设置 / 导入设置 |
| 日志 | 按钮 | 导出日志 |
| 重置所有设置 | 文字按钮 | 红色，底部右对齐，危险操作 |

### 9.3 各分类折叠开关

将"折叠上下文菜单"从通用设置移除，改为每个分类独立控制：

| 分类 | 开关名称 | 存储键 |
|------|----------|--------|
| Apps | 折叠应用菜单 | `foldAppsMenu` |
| Actions | 折叠动作菜单 | `foldActionsMenu` |
| New File | 折叠新建文件菜单 | `foldNewFileMenu` |
| Common Dir | 折叠常用文件夹菜单 | `foldCommonDirMenu` |

### 9.4 实施文件清单

| 文件 | 操作 | 说明 |
|------|------|------|
| `GeneralSettingsTabView.swift` | 重写 | 采用新分组卡片风格 |
| `AppsSettingsTabView.swift` | 修改 | 添加"折叠应用菜单"开关 |
| `ActionSettingsTabView.swift` | 修改 | 添加"折叠动作菜单"开关 |
| `NewFileSettingsTabView.swift` | 修改 | 添加"折叠新建文件菜单"开关 |
| `CommonDirsSettingTabView.swift` | 修改 | 添加"折叠常用文件夹菜单"开关 |
| `AppState.swift` | 修改 | 添加 4 个新的 `@AppStorage` 属性 |

### 9.5 设计规范

1. **分组风格**: 使用 `Form` + `Section` 结构，每组有独立标题
2. **图标状态**: 使用 SF Symbols 系统图标（如 `checkmark.circle.fill` 表示已授权）
3. **开关位置**: 每个分类列表顶部或 header 区域
4. **数据存储**: 使用 `@AppStorage` 持久化存储折叠状态
5. **文案规范**: 所有文案使用中文，"RClick"替换为"RClick"

- [ ] 通用设置界面与 macOS 系统设置风格一致
- [ ] 折叠开关从通用设置中移除
- [ ] 每个分类有独立的折叠开关
- [ ] 所有开关状态持久化保存
