| 6 | 外接磁盘动态监听 | 只在 `init()` 静态设置 | P1 - 需要修复 | ❌ 待修复 |

**修复计划**：参见 `重构计划.md` Phase 0。

**关于 MenuSnapshot 持久化的说明**：

原设计要求用 `@AppStorage` 持久化 `MenuSnapshot`，但这会引入问题：
- 业务数据（Apps、Actions、NewFiles、CommonDirs）已用 SwiftData 持久化
- 额外持久化快照会造成**双写一致性风险**
- 快照只是 IPC 消息封装，不是独立业务实体

**修正方案**：
- 快照从 SwiftData **实时构建**（SwiftData 查询毫秒级）
- 版本检查改用 **时间戳** 替代递增版本号
- 无需额外持久化存储

## 一、执行摘要

本重构计划的核心目标是构建**瘦 Extension 架构**，将 RClick 的 FinderSync 扩展简化为纯渲染层，所有业务逻辑和数据管理集中在主程序中。

### 核心原则（一句话）

> **FinderSyncExt 只负责 3 件事：渲染菜单、发送点击事件、全盘监听。所有业务逻辑在主程序中实现。**

### 点击事件数据结构

## 二、当前问题总结

### 2.1 架构层面

- **重复实现**: Extension 有 `IconCache`，Main App 有 `IconCacheManager`
- **数据模型冗余**: RCAction/ActionMenuItem、OpenWithApp/AppMenuItem 需要转换
- **通信协议简陋**: `MessagePayload.action` 使用 String 类型，缺乏类型安全
- **转换层复杂**: 需要在 SwiftData 模型和 MenuItem 模型间转换

### 2.2 性能问题

- **频繁 JSON 编解码**: 每次菜单更新都需要通过 UserDefaults 编解码
- **Tag 管理低效**: 使用随机数生成 tag，可能冲突导致性能问题
- **无缓存机制**: 图标每次都重新加载

### 2.3 数据一致性

- **同步延迟**: 配置更新需要等待通知和 UserDefaults 刷新
- **转换错误风险**: 模型转换可能丢失数据或引入 bug

### 2.4 用户体验

- **Main App 依赖**: Extension 在 Main App 关闭时无法工作
- **更新延迟**: 配置更新实时性差（依赖通知 + 数据转换）

## 三、设计目标

### 3.1 核心原则

1. **简洁性**: 减少重复代码，统一 Icon 缓存实现
2. **类型安全**: 使用枚举替代字符串，编译时检查
3. **性能**: 优化 Tag 管理，使用递增 ID；图标预加载缓存
4. **可靠性**: 错误处理完善，日志记录清晰

### 3.2 架构选择：瘦 Extension

Extension 只做 3 件事：

1. **接收消息** - 从主程序接收菜单配置、刷新指令
2. **渲染菜单** - 根据收到的配置渲染菜单（不读任何数据源）
3. **发送事件** - 用户点击后发送操作数据给主程序

**所有业务逻辑在主程序完成。**

### 4.4 IPC 消息安全设计

**威胁模型**：
- 恶意应用可能伪造 `DistributedNotificationCenter` 消息
- 消息可能被截获导致路径信息泄露
- 重复消息可能导致意外执行

**安全加固措施**：

#### 1. HMAC 消息签名

```swift
// Shared/MessageSecurity.swift
import CryptoKit
import Foundation

public class MessageSecurity {
    private static let sharedKey = "RClick_IPC_SharedKey_2026" // 应存储在 Keychain
    
    /// 为消息载荷添加 HMAC 签名
    public static func sign<T: Codable>(_ payload: T) throws -> SignedPayload<T> {
        let data = try JSONEncoder().encode(payload)
        let key = SymmetricKey(data: sharedKey.data(using: .utf8)!)
        let hmac = HMAC<SHA256>.authenticationCode(for: data, using: key)
        
        return SignedPayload(
            payload: payload,
            signature: Data(hmac).base64EncodedString()
        )
    }
    
    /// 验证消息签名
    public static func verify<T: Codable>(_ signed: SignedPayload<T>) -> Bool {
        guard let signatureData = Data(base64Encoded: signed.signature) else {
            return false
        }
        
        let key = SymmetricKey(data: sharedKey.data(using: .utf8)!)
        let expectedHMAC = HMAC<SHA256>.authenticationCode(
            for: try! JSONEncoder().encode(signed.payload),
            using: key
        )
        
        return signatureData == Data(expectedHMAC)
    }
}

public struct SignedPayload<T: Codable>: Codable {
    public let payload: T
    public let signature: String
}
```

#### 2. 发送者身份验证

```swift
// 验证发送者是否为 RClick 自身（代码签名验证）
```

### 4.5 新架构职责划分

#### Main App 职责

- **数据管理**: 读取 SwiftData，构建菜单配置
- **业务逻辑**: 处理所有动作执行（复制路径、删除、打开应用、创建文件等）
- **配置推送**: 数据变更时发送完整菜单配置给 Extension
- **事件处理**: 接收 Extension 的点击事件，根据场景 + 目录 + 选中项 + 菜单项信息执行操作
- **权限请求**: 请求完全磁盘访问权限（用户手动授权）

#### Extension 职责

- **FinderSync 监听**：设置 `directoryURLs`（默认 `/Users/` + 外接磁盘）
- **消息监听**：监听来自主程序的配置消息
- **菜单渲染**：根据配置渲染 NSMenuItem（纯渲染，不读取任何数据）
- **事件转发**：用户点击后发送事件给主程序，包含：
  - `scene`: 触发场景（右键物品/右键容器/工具栏/侧边栏）
  - `currentDirectory`: 当前目录路径
  - `selectedItems`: 选中的文件/目录列表
  - `menuItem`: 点击的菜单项信息（type, id, tag）
- **心跳发送**：每 10 秒发送 `heartbeat` 给主程序（让主程序知道 Extension 存活）

### 4.3 FinderSync 监听范围

**Extension 启动时设置监听目录**：

```swift
// FinderSyncExt.swift
override init() {
    super.init()

    // 默认监听 /Users/
    let usersDir = URL(fileURLWithPath: "/Users/")

    // 检测并添加外接磁盘
    let volumeURLs = FileManager.default.mountedVolumeURLs(
        includingResourceValuesForKeys: [.isBrowsableKey],
        options: [.skipHiddenVolumes]
    ) ?? []

    var allURLs = [usersDir]
    for volume in volumeURLs {
        if volume.path != "/" {  // 排除根目录
            allURLs.append(volume)
        }
    }

    FIFinderSyncController.default().directoryURLs = Set(allURLs)

    logger.info("Monitoring directories: \(allURLs.map { $0.path })")

    setupMessageHandlers()
    sendHeartbeat()  // 开始发送心跳
    
    // 注册磁盘挂载/卸载事件监听
    setupVolumeChangeObserver()
}

// 动态监听磁盘挂载/卸载
private func setupVolumeChangeObserver() {
    NotificationCenter.default.addObserver(
        self,
        selector: #selector(volumeDidChange),
        name: NSWorkspace.volumeDidMountNotification,
        object: nil
    )
    NotificationCenter.default.addObserver(
        self,
        selector: #selector(volumeDidChange),
        name: NSWorkspace.volumeDidUnmountNotification,
        object: nil
    )
}

@objc private func volumeDidChange(_ notification: Notification) {
    // 重新检测外接磁盘并更新监听目录
    DispatchQueue.main.asyncAfter(deadline: .now() + 1.0) { [weak self] in
        self?.updateMonitoringDirectories()
    }
}

private func updateMonitoringDirectories() {
    let volumeURLs = FileManager.default.mountedVolumeURLs(
        includingResourceValuesForKeys: [.isBrowsableKey],
        options: [.skipHiddenVolumes]
    ) ?? []
    
    var allURLs = [URL(fileURLWithPath: "/Users/")]
    for volume in volumeURLs {
        // 排除根目录和网络卷
        if volume.path != "/" && !isNetworkVolume(volume) {
            allURLs.append(volume)
        }
    }
    
    FIFinderSyncController.default().directoryURLs = Set(allURLs)
    logger.info("Updated monitoring directories: \(allURLs.map { $0.path })")
}

private func isNetworkVolume(_ url: URL) -> Bool {
    // 检测是否为网络卷（AFP、SMB、NFS 等）
    let values = try? url.resourceValues(forKeys: [.volumeIsNetworkKey])
    return values?.volumeIsNetwork == true
}
```

#### 监听策略

| 类型 | 处理方式 |
|------|----------|
| **默认监听** | `/Users/`（用户主目录） |
| **外接磁盘** | 自动检测并添加所有可浏览的挂载卷 |
| **网络卷** | 不监听（通过 `volumeIsNetwork` 排除） |
| **动态插拔** | 通过 `NSWorkspace` 通知动态更新监听目录 |
| **手动刷新** | 设置界面提供"刷新监听目录"按钮，若新磁盘右键菜单未出现可手动触发 |

> ⚠️ **注意**：`FIFinderSyncController.default().directoryURLs` 的设置对已加载的 Finder Sync 扩展可能不会立即生效，有时需要重启 Finder 才能覆盖所有情况。因此提供手动刷新按钮作为补充。

### 5.1 完全磁盘访问权限多点验证设计（P0 - 必须实施）

**问题**：单一检测点（如访问 `/Library/Application Support`）可能产生误判。

**解决方案**：使用三个检测点交叉验证

```swift
// RClick/Shared/PermissionChecker.swift
import Foundation

public class PermissionChecker {
    /// 多点验证完全磁盘访问权限
    /// 返回 true 表示可能已授权，false 表示可能未授权
    public static func hasFullDiskAccess() -> Bool {
        let testPaths = [
            "/Library/Application Support",
            "~/Library/Application Support",
            "/Library/Preferences"
```

## 八、附录：Extension 重连方案

### 8.1 问题背景

FinderSync Extension 由 macOS 系统控制加载时机，可能出现**主 App 已打开但 Extension 未加载**的情况，导致右键菜单不显示。

### 8.2 核心设计原则

| 原则 | 说明 | 违反后果 |
|------|------|----------|
| **Notification ≠ 状态** | 通知只能当"信号"，不能当"真相" | 丢消息=状态错乱 |
| **单一持久化源** | 业务数据只存 SwiftData，快照实时构建 | 双写一致性风险 |
| **Extension 必须支持主动拉取** | 不能只靠推送 | 无法自愈 |
| **所有通信必须幂等** | 重复消息不能出错 | 重复执行风险 |
| **时间戳防乱序** | 用时间戳判断配置新旧，拒绝旧配置 | 乱序覆盖问题 |

### 8.3 架构：推 + 拉混合模式

```mermaid
graph LR
    subgraph Before["纯推送（之前）"]
        A1[Main App] -->|发送消息| A2[Extension]
        A2 -.->|靠运气接收| A1
    end

    subgraph After["推拉混合（升级）"]
        B1[Main App] <-->|发送/请求| B2[Extension]
        B2 -.->|可主动拉取| B1
    end
```

### 8.4 四层防护策略

| 层级 | 机制 | 作用 | 优先级 |
|------|------|------|--------|
| L1 | **实时构建配置** | 从 SwiftData 实时构建，无持久化一致性风险 | P0 |
| L2 | **running 重试** | 主 App 启动后定期发送 running（携带当前配置） | P1 |
| L3 | **主动请求** | Extension 启动/缓存为空时主动请求 | P0 |
| L4 | **心跳检测** | Extension 定期发送 heartbeat，主 App 检测超时 | P1 |
| L5 | **手动刷新** | 用户点击菜单栏"刷新 Extension"按钮 | P2 |

**心跳方向**：

| 方向 | 消息类型 | 频率 | 目的 |
|------|----------|------|------|
| Main → Ext | `running`（携带 MenuConfigPayload） | 启动时 + 重试 3-4 次（指数退避：2s→4s→8s） | 告知 Extension 主程序已启动 |
| Ext → Main | `heartbeat` | 每 10 秒 | 告知主程序 Extension 存活 |
| Ext → Main | `requestConfig` | 启动/缓存为空时 | 请求菜单配置 |

**参数调整说明**：
- **重试次数**：从 6 次减少到 3-4 次，减少用户等待时间
- **重试间隔**：采用指数退避（2s, 4s, 8s），总时长从 30 秒降至 14 秒
- **超时阈值**：从 15 秒调整为 25-30 秒（心跳间隔的 2.5-3 倍），避免误判

### 8.5 能力评估

| 场景 | 之前 | 升级后 |
|------|------|--------|
| Extension 随机重启 | ❌ | ✅ |
| Notification 丢失 | ❌ | ✅ |
| 主 App 先启动 | ⚠️ | ✅ |
| Extension 后启动 | ⚠️ | ✅ |
| Finder 延迟加载 | ❌ | ✅ |
| 状态一致性 | 🟡 弱 | 🟢 接近强一致 |

**场景说明**：

1. **Extension 随机重启**：Finder 可能导致 Extension 进程重启，重启后缓存丢失
   - 解决：Extension 启动时主动发送 `requestConfig`

2. **Notification 丢失**：`DistributedNotificationCenter` 消息可能丢失
   - 解决：主 App 发送 `running` 后指数退避重试（2s→4s→8s）

3. **主 App 先启动**：主 App 启动时 Extension 尚未加载
   - 解决：主 App 发送 `running` 消息，Extension 后加载时主动请求

4. **Extension 后启动**：Extension 在 Finder 启动后才加载
   - 解决：Extension 启动时主动请求配置

5. **Finder 延迟加载**：Finder 可能延迟加载 Extension
   - 解决：主 App 重试发送 `running` 消息

6. **心跳超时检测**：主 App 25-30 秒未收到 `heartbeat`，标记 Extension 离线
   - 解决：用户可点击"刷新 Extension"按钮重新连接，或等待自动重连

### 8.6 菜单配置同步设计（修正版）

> **设计修正**：原方案要求持久化快照，但业务数据已用 SwiftData，无需额外持久化。

**核心原则**：
- **单一持久化源**：业务数据只存 SwiftData，避免双写一致性风险
- **实时构建快照**：每次请求时从 SwiftData 查询构建（毫秒级）
- **时间戳防乱序**：用时间戳替代递增版本号

**修正后的方案**：

```swift
// RClick/RClickApp.swift
class AppDelegate: NSObject, NSApplicationDelegate {
    // 内存缓存（仅用于减少重复构建）
    private var cachedMenuConfig: MenuConfigPayload?
    private var lastBuildTime: TimeInterval = 0
    
    /// 构建菜单配置（从 SwiftData 实时构建）
    func buildMenuConfig() -> MenuConfigPayload {
        let now = Date().timeIntervalSince1970
        
        // 如果 1 秒内已构建过，直接返回缓存（防抖）
        if now - lastBuildTime < 1.0, let cached = cachedMenuConfig {
            return cached
        }
        
        // 从 SwiftData 读取数据
        let actions = appState.actions.map { $0.toActionMenuItem() }
        let apps = appState.apps.map { $0.toAppMenuItem() }
        let newFiles = appState.newFiles.map { NewFileMenuItem(...) }
        let commonDirs = appState.cdirs.map { CommonDirMenuItem(...) }
        
        let config = MenuConfigPayload(
            timestamp: now,
            actions: actions,
            apps: apps,
            newFiles: newFiles,
            commonDirs: commonDirs
        )
        
        cachedMenuConfig = config
        lastBuildTime = now
        
        return config
    }
    
    /// 发送 running 消息（携带当前配置）
    func sendRunningMessage() {
        let config = buildMenuConfig()
        messager.sendToExtension(.running, data: config)
    }
    
    /// 响应 Extension 的配置请求
    func handleConfigRequest() {
        let config = buildMenuConfig()
        messager.sendMenuConfig(config)
    }
}

// MenuConfigPayload 增加时间戳
struct MenuConfigPayload: Codable {
    let timestamp: TimeInterval  // 构建时间戳
    let actions: [ActionMenuItem]
    let apps: [AppMenuItem]
    let newFiles: [NewFileMenuItem]
    let commonDirs: [CommonDirMenuItem]
    // ... 其他字段
}
```

**Extension 端版本检查**：

```swift
// FinderSyncExt.swift
class FinderSyncExt: FIFinderSync {
    private var currentTimestamp: TimeInterval = 0
    private var cachedMenuConfig: MenuConfigPayload?
    
    private func handleMenuConfig(_ data: Data?) {
        guard let config = decode<MenuConfigPayload>(data) else { return }
        
        // 时间戳检查：只接受更新的配置
        guard config.timestamp > currentTimestamp else {
            logger.info("Ignoring outdated config (timestamp=\(config.timestamp))")
            return
        }
        
        currentTimestamp = config.timestamp
        cachedMenuConfig = config
        logger.info("Menu config updated (timestamp=\(config.timestamp))")
    }
}
```

**优势对比**：

| 方案 | 原方案（@AppStorage） | 修正方案（SwiftData） |
|------|----------------------|----------------------|
| 持久化 | 双写（SwiftData + UserDefaults） | 单写（SwiftData） |
| 一致性 | 有风险 | 无风险 |
| 复杂度 | 需维护版本号、快照同步 | 只需时间戳 |
| 性能 | 快照可能过大 | 实时构建毫秒级 |
| 崩溃恢复 | 需恢复版本号 | 无需额外恢复 |

### 8.7 时间戳防乱序机制

**问题**：消息可能乱序到达，Extension 需判断是否接受。

**解决方案**：时间戳 + 配置哈希双重校验

```swift
struct MenuConfigPayload: Codable {
    let timestamp: TimeInterval      // 构建时间戳
    let configHash: String           // 配置内容 SHA256（可选）
    let actions: [ActionMenuItem]
    // ...
}

// Extension 端检查逻辑
func shouldAcceptConfig(_ new: MenuConfigPayload) -> Bool {
    // 首先检查时间戳
    if new.timestamp > currentTimestamp {
        return true
    }
    
    // 时间戳相同或更旧时，检查配置哈希
    // 如果哈希不同说明配置变化了，可能需要接受
    if new.configHash != currentConfigHash {
        logger.warning("Config hash mismatch, may need manual refresh")
        // 不自动接受，等用户手动刷新或下次更新
    }
    
    return false
}
```

```swift
// 检测脚本中的危险命令
func detectDangerousCommands(_ scriptContent: String) -> [String] {
    let dangerousPatterns = [
        "rm -rf /",
        "rm -rf ~",
        "sudo",
        "dd if=/dev/zero",
        ":(){ :|:& };:",  // Fork bomb
        "> /dev/disk",
        "mkfs",
        "newfs"
    ]
    
    return dangerousPatterns.filter { scriptContent.contains($0) }
}

// 使用示例
```

#### 3. 脚本执行沙盒（可选增强）

```swift
        (version 1)
        (deny default)
        (allow process-exec)
        (allow file-read* (subpath "/Users"))
        (allow file-write* (subpath "/Users"))
        (deny network-outbound)
        (deny system-ipc)
        """
    
    let process = Process()
    process.executableURL = URL(fileURLWithPath: "/usr/bin/sandbox-exec")
    process.arguments = ["-f", sandboxProfile, "/bin/bash", "-c", script, "--"] + files
    try process.run()
}
```

#### 4. 执行超时控制（后台线程执行）

> ⚠️ **重要**：脚本执行必须在后台线程调用，`process.waitUntilExit()` 如果在主线程执行会阻塞主程序界面。

```swift
// 脚本执行超时自动终止（必须在后台线程调用）
func executeScriptWithTimeout(_ script: String, files: [String], timeout: TimeInterval = 30) async throws {
    // 使用 async/await 确保在后台执行
    try await withCheckedThrowingContinuation { continuation in
        DispatchQueue.global(qos: .userInitiated).async {
            let process = Process()
            process.executableURL = URL(fileURLWithPath: "/bin/bash")
            process.arguments = ["-c", script, "--"] + files
            
            let pipe = Pipe()
            process.standardOutput = pipe
            process.standardError = pipe
            
            do {
                try process.run()
                
                // 启动超时监控
                DispatchQueue.global().asyncAfter(deadline: .now() + timeout) {
                    if process.isRunning {
                        process.terminate()
                        logger.warning("Script execution timed out after \(timeout)s")
                    }
                }
                
                process.waitUntilExit()  // 在后台线程阻塞是安全的
                let outputData = try pipe.fileHandleForReading.readToEnd()
                let output = String(data: outputData ?? Data(), encoding: .utf8) ?? ""
                
                if process.terminationStatus == 0 {
                    continuation.resume(returning: ())
                } else {
                    continuation.resume(throwing: ScriptError.executionFailed(output))
                }
            } catch {
                continuation.resume(throwing: error)
            }
        }
    }
}
```

#### 5. 脚本存储安全

```swift
// 脚本文件存储权限控制
func saveScriptToFile(_ content: String, to url: URL) throws {
    // 写入沙盒内目录
    let sandboxURL = FileManager.default.urls(for: .applicationSupportDirectory, in: .userDomainMask)[0]
        .appendingPathComponent("Scripts", isDirectory: true)
    
    try FileManager.default.createDirectory(at: sandboxURL, withIntermediateDirectories: true)
    
    let scriptFileURL = sandboxURL.appendingPathComponent(UUID().uuidString + ".sh")
    try content.write(to: scriptFileURL, atomically: true, encoding: .utf8)
    
    // 设置只读权限（chmod 500）
    let chmod = Process()
    chmod.executableURL = URL(fileURLWithPath: "/bin/chmod")
    chmod.arguments = ["500", scriptFileURL.path]
    try chmod.run()
}
```
